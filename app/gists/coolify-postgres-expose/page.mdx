When you add a Postgres database through a Coolify **Service Stack**, the container runs but its port is not exposed outside the stack's internal Docker network by default. Any service outside that stack - including other Coolify-managed apps on the same host - cannot reach it.

## The symptom

Your app logs a connection refused or timeout error even though the Postgres container shows as running in the Coolify UI. The database is healthy; it just isn't reachable from outside its stack network.

## The fix

In the Coolify UI, open the Postgres service inside your stack and find the **Ports** section. Enable **Expose Port to Public** and set the host port you want to bind to (e.g. `5432` or a custom port if `5432` is already taken on the host).

```
Expose Port to Public: enabled
Host Port: 5432
Container Port: 5432
```

Save and redeploy the service. Coolify will add a `ports` binding to the underlying Docker Compose config, making the container reachable at `host-ip:5432` from any other container or service on the same machine.

## Connecting from another Coolify service

Use the host's internal Docker bridge IP (`172.17.0.1` by default on Linux) or the host's LAN IP as the database host in your app's environment variables:

```
DATABASE_URL=postgresql://user:password@172.17.0.1:5432/dbname
```

Avoid using `localhost` - from inside a container, that resolves to the container itself, not the host.

## Note on security

Exposing a port to the host makes it reachable from anywhere that can reach the host machine. If the host is behind Cloudflare Tunnel with no open inbound ports, the risk is low. Otherwise, bind Postgres to a specific interface or add a firewall rule to restrict access to trusted IPs.
